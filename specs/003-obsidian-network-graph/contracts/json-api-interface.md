# JSON API Interface Contract

**Interface**: Network Graph Data Endpoint
**Path**: `/data/network-graph.json`
**Method**: GET
**Content-Type**: `application/json`

## Contract Overview

This endpoint provides structured graph data for the network visualization, generated at build-time by Eleventy from blog post frontmatter.

## Response Schema

### Success Response (200)

```json
{
  "nodes": Node[],
  "links": Link[],
  "metadata": Metadata
}
```

### Node Schema

```typescript
interface Node {
  id: string;           // Unique identifier (post slug or tag name)
  type: "post" | "tag"; // Node type for visual differentiation
  title?: string;       // Post title (posts only)
  name?: string;        // Tag display name (tags only)
  url?: string;         // Navigation URL (posts only)
  date?: string;        // ISO date YYYY-MM-DD (posts only)
  tags?: string[];      // Associated tag names (posts only)
  count?: number;       // Connected post count (tags only)
}
```

### Link Schema

```typescript
interface Link {
  source: string;                    // Source node ID
  target: string;                    // Target node ID
  type: "post-tag" | "post-post";   // Relationship type
  strength: number;                  // Connection strength (1.0-2.0)
  sharedTags?: string[];            // Common tags (post-post only)
}
```

### Metadata Schema

```typescript
interface Metadata {
  generatedAt: string;     // ISO timestamp of data generation
  nodeCount: number;       // Total node count
  linkCount: number;       // Total link count
  postCount: number;       // Post node count
  tagCount: number;        // Tag node count
}
```

## Contract Guarantees

### Data Integrity
- All `source` and `target` IDs in links reference existing nodes
- Post nodes always have `title` and `url` fields
- Tag nodes always have `name` and `count` fields
- No duplicate node IDs within the dataset
- Link strengths are always positive numbers

### Performance Characteristics
- Response size typically 50-500KB for standard blog scale
- Static file served directly (no server-side processing)
- Cacheable with standard HTTP cache headers
- Generated at build-time (no runtime database queries)

### Backward Compatibility
- New optional fields may be added to nodes/links
- Existing field names and types will not change
- `type` field values ('post', 'tag', 'post-tag', 'post-post') are stable
- Schema structure (nodes array, links array, metadata object) is stable

## Client Usage Pattern

```javascript
// Standard fetch pattern for graph data
async function loadGraphData() {
  try {
    const response = await fetch('/data/network-graph.json');
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    const data = await response.json();

    // Validate required fields
    if (!data.nodes || !data.links) {
      throw new Error('Invalid graph data structure');
    }

    return data;
  } catch (error) {
    console.error('Failed to load graph data:', error);
    // Implement fallback behavior
    return { nodes: [], links: [], metadata: {} };
  }
}
```

## Build-Time Generation

This endpoint is generated by Eleventy using a collection that:
1. Processes all Markdown files in `src/posts/`
2. Extracts frontmatter (title, tags, references)
3. Creates node and link structures
4. Outputs JSON to the specified permalink

The generation process ensures data consistency and validates all relationships at build-time, eliminating the possibility of runtime data integrity errors.
